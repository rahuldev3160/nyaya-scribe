# BUG-016 — Jinja2 `.items` resolves to Python dict builtin, not the key

**Status:** FIXED (root cause addressed S20)
**Severity:** HIGH
**Category:** template
**First seen:** S18 (`/rbi/prep` → 500)
**Recurred:** S20 (`/rbi` → 500 in newly-written `rbi_dashboard.html`)
**Pattern:** JINJA2-001

---

## Root Cause

Jinja2 attribute access (`object.attr`) first checks the object for a Python attribute with that name. Python dicts have a builtin method `.items()` — so `sec.items` resolves to the method object, not the dict key named `"items"`. Iterating over a method object → `TypeError: 'builtin_function_or_method' object is not iterable`.

This is invisible in Python (where `d['items']` and `d.items()` behave as expected) but surfaces in Jinja2 because Jinja2 uses dot-notation for both attribute access and key lookup.

---

## Manifestation

```
TypeError: 'builtin_function_or_method' object is not iterable
  File "rbi_prep.html", line 51 — {% for name, value, note, verify in section['items'] %}
  File "rbi_dashboard.html", line 84 — {% for item in sec.items %}
```

---

## What was done at S18 (incomplete fix)

- Changed `section.items` → `section['items']` in `rbi_prep.html`
- Did NOT rename the key at the data layer
- Did NOT check other templates for the same pattern
- Did NOT write this bug record
- Did NOT add JINJA2-001 to PATTERNS.md

This left the collision in place — any new template written with the same dict shape and dot-notation would re-hit the same crash. Which it did in S20 (`rbi_dashboard.html`).

---

## Root Fix (S20)

Renamed the dict key `"items"` → `"rows"` at the data layer in both blueprints:

- `rbi_prep_bp.py` — `KEY_SECTIONS` constant: all 6 section dicts `"items": [` → `"rows": [`
- `rbi_dashboard_bp.py` — `_sec_map` builder: `"items": []` and `["items"]` → `"rows": []` and `["rows"]`

Updated template references:

- `rbi_prep.html` line 51: `section['items']` → `section['rows']`
- `rbi_dashboard.html` line 84: `sec['items']` → `sec['rows']`

With `"rows"` as the key, dot-notation `sec.rows` is safe — no Python dict builtin collision.

---

## Prevention Rule

**Never name a dict key with the same name as a Python dict method** (`items`, `keys`, `values`, `get`, `update`, `pop`, `clear`, etc.) when that dict will be passed to a Jinja2 template. Use names like `rows`, `entries`, `data_items`, `members` instead.

See: JINJA2-001 in `~/.claude/knowledge/patterns/PATTERNS.md`
