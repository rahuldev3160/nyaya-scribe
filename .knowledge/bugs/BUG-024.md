# BUG-024 — Orphaned user_events + ghost sessions on deleted users

**Status:** FIXED  
**Severity:** MEDIUM  
**Category:** data-integrity  
**Session:** S29  
**Commit:** TBD  

---

## Summary

Two independent gaps let `user_events` rows be written with no corresponding `users` row:

1. **No FK constraint on `user_events.user_id`** — SQLite allowed inserts referencing non-existent users. `PRAGMA foreign_keys=ON` was set on every connection, but the table had no `REFERENCES` clause so the pragma had nothing to enforce.

2. **`validate_session` only checked `sessions` table** — a valid session token for a user whose `users` row was deleted (e.g. during a DB reset in dev) still returned a `user_id`. Events then logged against that user_id produced orphan rows.

Observed impact: 1 real visitor on 2026-06-03 (`d87e85b1`) left a `page_time` event in `user_events` with no matching row in `users`. No email or profile data was recoverable for them. 3 total orphaned events were present (d87e85b1 + test-user-s15).

---

## Root Cause

- `user_events` DDL never included `REFERENCES users(user_id)` so the FK pragma was a no-op for that table.
- `validate_session` queried `SELECT … FROM sessions WHERE session_token=?` — sessions table has a FK to users, but after a DB reset the sessions table was also empty; the orphaned event pre-dated the reset so it carried a stale user_id with no session or user row.
- Most likely cause of the specific orphan: `nyaya.db` was reset during S25 dev work, clearing users and sessions but not user_events (or the event predated nyaya.db setup entirely).

---

## Fix

### 1. `web/auth.py` — `validate_session`

Added `JOIN users u ON s.user_id = u.user_id` to the query. Now returns `None` (and kills the cookie) for any session whose user row no longer exists.

```python
row = conn.execute(
    """SELECT s.user_id, s.expires_at
       FROM sessions s
       JOIN users u ON s.user_id = u.user_id
       WHERE s.session_token=?""",
    (token,)
).fetchone()
```

### 2. `web/app.py` — `_run_nyaya_migrations()`

Added idempotent migration that recreates `user_events` with FK:

```sql
CREATE TABLE user_events_new (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    ...
);
INSERT INTO user_events_new SELECT * FROM user_events WHERE user_id IN (SELECT user_id FROM users);
DROP TABLE user_events;
ALTER TABLE user_events_new RENAME TO user_events;
```

Guard: checks for `REFERENCES users` in `sqlite_master` before running — idempotent on restarts.

### 3. `seeds/nyaya_seed.db`

Same FK applied to the seed schema so fresh installs never have the gap.

### 4. Live `data/nyaya.db`

Migration applied directly — 3 orphaned events dropped, 40 remain, all with valid `users` FK.

---

## Prevention

- Any new table that writes per-user data must include `REFERENCES users(user_id) ON DELETE CASCADE` (or `SET NULL` for soft-delete scenarios).
- `validate_session` now fails safe: deletes the stale cookie on next request.
- `_run_nyaya_migrations()` is called at every app start and is idempotent — future nyaya.db schema changes should follow the same pattern.
